Today I'm going to talk about a new CSS Framework that I created called Scales. You might be saying to yourself:

CSS is easy!

Why would I need a framework?

You're right. CSS is easy. But that's exactly the reason, the simplicity, the looseness, that makes a solid architecture the best way to manage CSS at any reasonable scale. A CSS framework can provide that architecture by controlling specificity, enforce naming conventions, manage source order, and keeping things consistent.
Clearly there is a need for css frameworks. There are dozens of them out there already. Some of them are really great. They help solve a lot of common problems, the biggest being a design out of the box for people who aren't great at design. Now you should have a new question:

Why would you write another framework?

What's wrong with existing frameworks?

  • Design
  • Tools
  • RWD assumptions
  • Tightly coupled components
  • BLOAT!

Let me make the decisions!

I started looking for a framework or UI toolkit to make development more consistent on client facing web projects. Wasn't happy with what I found. They all make too many decisions for me.
  • A design I would have to override
  • I want to make these choices. Grids, icons, etc.
  • Predefined breakpoints, not content first
  • Requiring me to use all or nothing
  • All of this adds bloat

A framework should help me solve design problems.

It shouldn't do my work for me.

I actually don't agree that most of these frameworks should be called frameworks. They do too much. Things like bootstrap and foundation should more accurately be called UI Toolkits because they actually make design decisions for you and give you more than just a helping hand with your CSS.

What makes Scales different?

A design-free, independent collection of OOCSS patterns, framework settings, and utilities for quickly building responsive web projects

Design Free

Like I mentioned before, I get a design from the design team or I make it up myself. A design isn't helpful for me and just means I have to override a bunch of styles.

Independent

Each piece of scales is broken up into it's own repo and is independent of the others. The only dependency some of them have is a clearfix utility.

Object Oriented CSS

CSS "Object" - A repeating visual pattern that can be abstracted into an independent, reusable snippet of HTML and CSS

OOCSS is at the heart of what Scales is. But what is a CSS object?

Media Object

A fixed piece of media on the left (image, video, icon) and some unknown content to the right.

This is the poster child for OOCSS methodology. A media object is made up of a fixed piece of media on the left (image, video, icon) and some unknown content to the right. These are seen all over the web on nearly every site. Lets look at a fb status as an example. See any media objects in there? There are 4 and they all could use this same markup pattern.

            .media {
        @extend %clearfix;
        display: block;
    }
    .media__figure {
        float: left;
        margin-right: 1em;

        > img {
            display: block;
        }
    }
    .media_body {
        overflow: hidden;
        display: block;

        &,
        > :last-child {
            margin-bottom: 0;
        }
    }
And just a tiny bit of CSS. Now multiply that over 1000 status messages in your newsfeed and you can see the CSS savings you get with this methodology.

OOCSS Principles

  • Separation of Structure from Skin

  • Separation of Content from Container

The OOCSS principles are simple.

Structure vs. Skin


            .btn {
        display: inline-block;
        font-size: 1.5em;
        padding: 1em 2em;
        text-align: center;
        border-radius: 5px;
        cursor: pointer;
    }
    .btn--primary {
        background-color: blue;
        color: white;
    }
    .btn--warning {
        background-color: orange;
        color: black;
    }
    

            
This is the main principle of OOCSS and it's really easy to do. Simply separate the structural elements of your object into one class (btn) and then extend the object to add a skin to it (btn--primary). So we can make any number of buttons that look different, but share the same CSS for the structure.

Content vs. Container


            .category-list h3 {
        font-size: 2em;
        color: red;
    }
    

            .heading--secondary {
        font-size: 2em;
        color: red;
    }
Imagine having a list of categories on your site that have a heading above them. We could do something like this. But we are essentially tieing that h3 and it's styles to the .category-list class. What if we want to use the same style heading somewhere else? For content object like headings, we would make a new class for the h3 instead. Now we can use that class anywhere for the same styles.

Yay Modularity!

(Seriously, OOCSS is awesome)

Yes, that's it!

You may be thinking "There doesn't seem to be that much here." Well, you're right. Scales is intentionaly a tiny framework. For those familiar with PHP frameworks, you can think of Scales like a micro framework, such as Slim. It only does a few things to help you get a leg up and makes it easy to extend what's there and add the tools that you want. Other frameworks like Bootstrap would be more comparable to a Full-stack framework like Zend. It does a TON of stuff for you, but it's really hard to change the tools it decided for you to use.

The Future of Scales

  • NuGet package
  • NPM packages
  • Yeoman generator
  • A prototyping theme
  • A theme based on the QL Digital Pattern Library
Right now Scales is only available as Bower packages. I will be adding a NuGet package for easier use in .NET projects, I'll make it available on NPM for those who don't like bower, and I'll be making a generator using Yeoman for easier project setup. On top of that, I will be building a theme on top of scales that will make prototyping really easy and the thing I'm most excited about is a theme based on the QL pattern library. Not a lot of people have seen this, but a few engineers on the web tier worked with the product design team to create guidelines for what QL websites should look like in the future.
