New DPL Sneak Peek

A few UI Engineers and UX Designers have been working on a new version of the DPL.

Current DPL

The soon to be deprecated DPL
  • Design recommendations
  • Basic patterns
  • No code
  • Very little real guidance

Most of you are probably aware of the current DPL. Good first step. Served it's purpose as a design recommendation for the basic patterns most apps uses. Failed to provide actual implementation details and had very little guidance on how and when to use each pattern.

A new DPL is coming!

The Alpha version of the new DPL
  • New pattern designs
  • More complex patterns
  • Code recommendations
  • Lots of documentation

Good news! The UX team audited the current DPL and have updated and clarified what we currently have. We are going to be adding more patterns beyond just the basic elements. We're adding implementation guidance. And we'll had much more documentation around how and when to use patterns, what not to do with certain patterns, accessibility guidelines, and anything else we can think to document.

Not just a pattern library anymore.

Global Namespace

Isolates DPL styles from application specific styles to avoid conflicts.

Class Type Namespace

More info from Harry Roberts at csswizardry.com
  • `o-`: Object. Used in any number of unrelated contexts to the one you can currently see it in. Dangerous to change css.
  • `c-`: Component. Implementation-specific piece of UI. Safe to change css.
  • `u-`: Utility. Highly specific. Highly reusable. Usually single purpose. High specificity.
  • `is-`, `has-`: State. Temporary, optional, short-lived. Usually applied or changed with JS.
  • `js-`: Javascript behavior. Signifies that JavaScript is bound to it. No CSS should be tied to these.

Provides clarity and confidence in what the purpose of the class is and how safe tehy are to change.

ITCSS

It's like Harry Roberts knows what he's talking about

ITCSS

  • Settings: Global stuff. Variables, colors, grid settings, etc. No CSS
  • Tools: Global Mixins and Functions No CSS
  • Generic: High level, far-reaching styles. Normalize, box-sizing, resets, etc. Rarely modified.
  • Base/Elements: Unclassed html elements. What do things look like without classes?
  • Objects: Highly reusable, abstract, design-free, mostly layout-driven styles. Think OOCSS objects like the Media Object. Shouldn't really be changed much because they could affect many things.
  • Components: Recognizable pieces of the UI. Safe to change their CSS.
  • Utilities: Highest specificity layer. Helper classes, overrides, etc. Often use !important here to make sure they really have the highest specificity.

System and Tooling

  • Built on top of Scales CSS
  • Scales DPL Theme
  • Drizzle

Basically built as a theme on top of Scales. The Scales DPL theme lives in its own repo so that it can be consumed by any application via NPM. Even the tool that the pattern library is built on just consumes the theme.

Drizzle is basically a static site generator for pattern libraries. It uses handlebars to make writing new patterns and variations easy. Uses frontmatter from Jekyll to allow for a bunch of meta data on a patterns that's displayed with the pattern.

Process

  • Add new pattern HTML and CSS in Drizzle.
  • Open a PR in Github Enterprise.
  • Code review and changes
  • Merge into master as In Progress
  • Once tested and proved out, move the styles from Drizzle to the Scales DPL Theme.
  • Some sort of version/tag/release process that doesn't currently exist.

Really is no process yet. Still being worked out. In general though, this is the process to follow if you want to contribute.

Design System Contributors

The centralized team can't do everything I've brought up so a larger virtual team is very important. This team is made up of everyone in the organization that is interested in using the system and growing it. It should involve contributions both internally to the system and externally as well as pairing between the design system team and people building our products.
